# Make the 3D reference window distance-based

Status: ready-for-agent
Category: enhancement

## Parent

.scratch/3d-web-replay-and-scenario-suite/PRD.md

## Triage Notes

**What we've established so far:**

- The current replay `local_plan` is the controller-facing reference window in world coordinates, not the MPPI optimal trajectory. The MPPI output remains `optimal_trajectory`.
- In the reproduced `narrow_gap_t_volume_3d` replay, frame 0 has 12 local-plan points from `x = 0.0` to `x = 0.3705263`, so the displayed window is only about 0.37 m long.
- `select_global_plan()` currently selects `nearest_idx + time_steps` points by index, not by physical distance.
- The narrow-gap reference path has 96 points over about 3.2 m, so a 12-point index window is too short to be visually meaningful and can sit inside the T-shaped 3D robot volume.
- The 3D optimizer scores `plan[:time_steps]`, so simply exporting more dense points is insufficient. The fixed-size controller-facing reference window itself needs to span a configurable physical lookahead distance.
- No matching `.out-of-scope/` rejection exists in this repo.

## Agent Brief

**Category:** enhancement
**Summary:** Select and display the 3D controller reference window by physical lookahead distance instead of raw reference-path point count.

**Current behavior:**
The 3D scenario runner selects the current local plan/reference window by taking `time_steps` consecutive points from the global reference path after the nearest point. For dense reference paths, this can cover less than the robot body length. In replay, the `Local plan` layer appears inside the robot and along the global reference path, which is technically a controller reference window but misleading as an inspection layer.

**Desired behavior:**
The controller-facing reference window should remain fixed-size for the MPPI controller, but its points should be sampled across a configurable physical lookahead distance. The replay/viewer naming should make the semantics clear: this layer is the reference window passed to the controller, not the MPPI optimal trajectory.

**Key interfaces:**

- Scenario config — add or use a 3D reference-window lookahead distance setting, with a sensible default for existing static scenarios.
- `select_global_plan()` / `select_local_plan()` — select or resample a fixed-size world-frame reference window over physical arc length, padding at the goal if needed.
- Controller input contract — preserve the controller-facing plan shape expected by MPPI (`time_steps` points) while increasing the physical span of those points.
- Replay frame schema — keep exporting the world-frame controller reference window, but either rename/add a clearer `reference_window` field or maintain backward-compatible `local_plan` while documenting the semantics.
- Passive Web 3D viewer — label the layer as `Reference window` or otherwise clarify that it is not the MPPI optimal trajectory.

**Acceptance criteria:**

- [ ] For `narrow_gap_t_volume_3d`, frame 0's exported controller reference window spans a configurable physical distance that clearly extends beyond the T-shaped 3D robot volume.
- [ ] The controller still receives a fixed-size plan compatible with `time_steps`.
- [ ] Dense reference paths no longer produce visually tiny controller reference windows.
- [ ] Replay data and/or viewer labeling distinguishes the controller reference window from `optimal_trajectory`.
- [ ] Tests cover a dense reference path where the selected reference window spans distance rather than just `time_steps` adjacent raw points.
- [ ] Existing replay viewer layers for reference path, executed path, optimal trajectory, and rollouts still work.

**Out of scope:**

- MPPI cost-function redesign or parameter tuning.
- Changing the meaning of `optimal_trajectory`.
- Browser-owned local-frame reconstruction or browser-owned planning.
- Dynamic obstacle behavior.
