## Agent Brief

**Category:** bug
**Summary:** Ensure the current local plan is clearly visible and inspectable in the HTML replay viewer.

**Current behavior:**
Replay data includes a per-frame local plan in global coordinates, and the viewer has code intended to render it. In manual HTML playback, the local plan is not visibly apparent to the user.

**Desired behavior:**
The local plan should be visually obvious when its layer is enabled. It should be distinguishable from the global reference path, executed path, optimal trajectory, and rollouts across common camera modes and during timeline scrubbing.

**Key interfaces:**

- Replay frame schema — `local_plan` remains a world-frame list of `[x, y, z, yaw]` states.
- Passive Web 3D viewer layer controls — the local-plan toggle should reliably show/hide the current frame's local plan.
- Viewer line rendering — local plan styling should remain stable and distinguishable even when it overlaps the reference path.

**Acceptance criteria:**

- [ ] Loading a `narrow_gap_t_volume_3d` replay shows the local plan when the local-plan layer is enabled.
- [ ] The local plan remains visually distinguishable from the global reference path and optimal trajectory.
- [ ] Timeline scrubbing updates the local plan to the selected frame.
- [ ] Turning the local-plan layer off hides the local plan without affecting other path layers.
- [ ] A viewer smoke/static test or documented manual check verifies the local-plan layer with representative replay data.

**Out of scope:**

- Changing local-plan selection logic in Python.
- Changing MPPI trajectory generation or cost functions.
- Browser-owned reconstruction of local-frame controller inputs.
