# PRD: MID-360-Like 3D Point Cloud Observation

Status: ready-for-agent

## Problem Statement

The current yaw-only 3D scenario runner does not model realistic perception. It uses Range-based local 3D observation: selecting nearby Legacy 3D obstacle points from a global truth point set, cropping by distance, and transforming them into the robot-local yaw-only frame. That is useful for the first 3D controller path, but it is effectively a local obstacle oracle.

The user wants the 3D controller to use a more realistic realtime 3D point cloud observation model. The model should be shaped like a Livox MID-360 class sensor: omnidirectional horizontal coverage, asymmetric vertical coverage, a blind range, and a conservative maximum range. It should generate observations from obstacle geometry, not from pre-sampled global obstacle points, and the controller should consume those true observations.

The replay viewer should also make the distinction visible: static obstacle geometry is the world truth, while the observed point cloud is what the sensor sees at each planning step.

## Solution

Replace the Range-based local 3D observation path in the 3D scenario runner with a MID-360-like 3D point cloud observation model. The sensor model raycasts a deterministic angular grid against authoritative obstacle geometry for each planning step, producing an Observed 3D point cloud in world coordinates. The controller receives those observed points transformed into the robot-local yaw frame and packed under its existing obstacle point budget.

The first version is deterministic and scoped: it supports axis-aligned box obstacle geometry, uses a configured conservative sensor range of `0.1m` to `10.0m`, uses MID-360-like FOV defaults of `360°` horizontal and `[-7°, +52°]` vertical, and does not model high-reflectivity range extension, elevation-dependent range curves, random noise, dropout, rolling scan timing, or motion distortion.

Update the replay schema so scene data contains authoritative obstacle geometry only, while each frame contains the current world-frame observed point cloud. Update the Passive Web 3D viewer to remove the old static obstacle points layer and add a dynamic observed cloud layer that changes with the timeline.

## Implementation Decisions

- Use the project glossary terms MID-360-like 3D point cloud observation, Observed 3D point cloud, Controller 3D obstacle points, Legacy 3D obstacle points, Yaw-only 3D motion, and Passive Web 3D viewer.
- Follow the ADR that records obstacle geometry raycast observations as the authority for MID-360-like 3D perception.
- Treat obstacle geometry as the authority for sensor generation, replay scene data, and collision metrics.
- Remove the new observation path's dependence on legacy global sampled obstacle points.
- Generate an Observed 3D point cloud per planning step by raycasting a MID-360-like sensor model against obstacle geometry.
- Represent the first observed cloud as world-frame points.
- Transform the observed cloud into the robot-local yaw frame before passing it to the 3D controller.
- Keep the 3D controller's current obstacle point input shape and fixed-size packing model.
- Reuse the original 2D controller packing semantics: when observed points exceed the configured budget, keep nearest points by local squared range and mask padded slots.
- Put sensor settings under a top-level `sensor` configuration section.
- Use `type: mid360_like` for the first sensor model.
- Use default sensor range `min_range_m: 0.1` and `max_range_m: 10.0`.
- Use default sensor FOV `horizontal_fov_deg: 360.0`, `vertical_min_deg: -7.0`, and `vertical_max_deg: 52.0`.
- Use deterministic regular angular-grid sampling for the first version.
- Configure angular-grid density through sensor sample counts.
- Treat each planning step's observed cloud as an integrated point cloud for that planning interval.
- Do not model per-ray timestamps, rolling scan timing, or motion distortion in the first version.
- Reserve noise, dropout, and seed fields in sensor configuration, but keep noise and dropout disabled by default.
- Do not model high-reflectivity range extension.
- Do not model elevation-dependent effective range curves.
- Support axis-aligned box obstacle geometry in the first raycast implementation.
- Do not support meshes, rotated boxes, spheres, cylinders, occupancy grids, OctoMap, or ESDF maps in this PRD.
- Implement nearest-hit raycast semantics so that occlusion is represented.
- Remove `simulation.observation_range` from new 3D scenario configs.
- Remove `simulation.max_obstacle_points` from new 3D scenario configs.
- Use sensor range settings for perception range.
- Use controller `max_obs_num` for controller point budget.
- Keep collision and clearance metrics based on obstacle geometry truth, not on observed point clouds.
- Update replay scene data to include obstacle geometry and omit legacy scene obstacle points.
- Update replay frame data to include the current world-frame observed point cloud.
- Do not include controller-packed local obstacle points in the base replay schema.
- Update the Passive Web 3D viewer to render a dynamic Observed cloud layer from frame data.
- Remove or replace the old static Obstacles point layer in the viewer.
- Keep static Obstacle geometry as a separate viewer layer.
- Make the observed cloud visually distinct from obstacle geometry.
- Preserve Python ownership of simulation, sensor generation, controller invocation, metrics, and replay export.
- Preserve the browser as a passive viewer that consumes replay data.

## Testing Decisions

- The primary test seam is the 3D scenario runner plus replay export. Tests should call the runner and replay builder to verify external behavior across sensor config, raycast observation, controller input source, metrics, and schema.
- The supporting test seam is a pure MID-360-like box raycast generator. Tests should verify geometric behavior, not internal loops or implementation details.
- The viewer test seam should follow the existing replay viewer smoke-test style, validating that the new replay schema can load and that the dynamic observed cloud layer exists.
- Raycast tests should verify that boxes outside `[-7°, +52°]` vertical FOV produce no observed points.
- Raycast tests should verify that boxes outside the configured `0.1m` to `10.0m` range produce no observed points.
- Raycast tests should verify that an empty obstacle geometry list produces an empty observed cloud.
- Raycast tests should verify nearest-hit occlusion when two boxes lie along the same ray.
- Runner tests should verify that controller inputs are derived from observed raycast points rather than legacy global obstacle points.
- Runner tests should verify that observed points are transformed into robot-local yaw coordinates before controller input.
- Controller packing tests should lean on existing 3D controller behavior and the original 2D prior art: over-budget points keep the nearest local points and mask padded slots.
- Replay schema tests should verify that `scene.obstacle_points` is absent.
- Replay schema tests should verify that `scene.obstacle_geometry` is present.
- Replay schema tests should verify that every frame contains world-frame `observed_point_cloud`.
- Replay schema tests should verify that replay data is finite and JSON-serializable without NaN.
- Metrics tests should verify that clearance and collision are based on obstacle geometry truth rather than observed point cloud contents.
- Viewer smoke tests should verify that static obstacle geometry and dynamic observed cloud layers render from representative replay data.
- Tests should run headlessly on CPU without ROS, Gazebo, RViz, GPU, or a live Web server.

## Further Notes

- The MID-360 public specifications used for this design are horizontal `360°` FOV, maximum vertical `59°` FOV, minimum detection range `0.1m`, and reflectivity-dependent maximum range. The design intentionally chooses a conservative configured `10.0m` maximum range and does not model high-reflectivity extension.
- The MID-360 user manual shows asymmetric vertical FOV and describes different effective range behavior across vertical FOV. This PRD captures the asymmetric `[-7°, +52°]` FOV but leaves vertical range variation out of scope.
- The existing 2D controller prior art for lidar point packing is the reference for 3D controller point-budget behavior.
- This PRD supersedes the new 3D scenario runner's use of Range-based local 3D observation. That term remains useful for describing the legacy model, not the target MID-360-like perception path.
